一、功能實用性 — 建議新增功能

以下功能類似「今日入住提醒」，都是對日常營運有直接幫助的實用性增強。按優先級排列。

P0 1. 總覽頁「今日入住 / 今日退房」提醒卡片
目前「今日入住提醒」只存在於 Telegram Bot（每日 09:00 推送），Web 總覽頁完全沒有。 建議在總覽頁 KPI 卡片下方加入「今日入住 N 筆」+「今日退房 N 筆」橫條卡片，點擊展開名單。 這樣不打開 Bot 也能一眼看到當日待辦。
bookings.js 已有 Utils.isToday() 工具函數可直接使用。
P0 2. 費用未填 / 利潤未填 提醒
已退房（checked-out）的訂房，如果 chargeGuest 或 chargeCompany 未填， 月結時容易遺漏。建議在總覽頁顯示「N 筆待收費」「N 筆待填利潤」警示徽章，點擊跳轉到費用/利潤頁。
P0 3. 待確認訂房超時提醒
pending 狀態超過 3 天未確認的訂房，可能遺漏或忘記。建議總覽頁顯示「N 筆待確認超時」紅色警示。 Bookings.getBookingsNeedingReminder() 已有類似邏輯可參考。
P1 4. 點擊代理名快速篩選
在任何表格中點擊代理名，自動跳轉到「代理業績」頁並篩選該代理。減少手動切換頁面+選下拉的操作。
P1 5. 確認號快速複製
點擊確認號欄位即可複製到剪貼簿，並顯示「已複製」Toast。減少在手機上長按選取的不便。
P1 6. 訂房重複檢測
新增訂房時，如果同一客人姓名 + 入住日期重疊，跳出警告「疑似重複訂房：XXX 已有 7/10-7/12 的訂房」。防範重複登記。
P1 7. 匯出月度 CSV 報表
一鍵匯出當月所有訂房明細為 CSV，方便傳給會計或老闆看。包含：客人/代理/酒店/日期/費用/利潤等欄位。
P2 8. 批次狀態更新
表格加 checkbox，勾選多筆後批次確認/取消。月底大批量操作時很實用。
P2 9. 代理月度對帳單
在代理業績頁加「匯出對帳單」按鈕，生成某代理某月的完整明細+合計，可列印或匯出。
P2 10. 最近操作日誌
記錄最近 50 筆操作（新增/編輯/刪除/狀態變更/封存），含操作人+時間，方便追溯爭議。

二、結構完整性

✓ 現有優勢

清晰的分層架構
core / data / sync / calc / ui / pages / bridge 七層分離，職責明確
IIFE 模組封裝
OverviewPage / FeesPage / ProfitPage 等模組避免全域污染
Event Bus 解耦
38 個事件讓模組間通信不直接依賴
純函數計算層
calc/stats + calc/filters 無副作用，可獨立測試

待改善項

P0 1. State.set() 觸發 UI_RENDER 過於頻繁
state.js 第 38 行：每次 set() 都 emit UI_RENDER， 而 app.js 監聽 UI_RENDER 會重渲染整個頁面。 同步下載時連續 set 6 次（bookings/hc/agentList/...），等於渲染 6 次。 建議：加入 debounce 或區分「數據變更」和「UI 刷新」事件。
P0 2. _hasChanged() 比較邏輯不嚴謹
watchers.js 第 362-374 行：只比較陣列長度和 _updatedAt 總和。 如果兩個不同的陣列碰巧長度相同且 _updatedAt 總和相同，會誤判為「無變化」而跳過更新。 建議：改用 JSON.stringify 比較，或至少比較每個元素的 _updatedAt 值。
P1 3. afterFirebaseReady 競態風險
app.js 第 245-257 行：同時調用 startWatchers() + syncDownloadAll()。 watchers 的即時 on('value') 和 syncDownloadAll 的 once('value') 會同時觸發， 兩者都嘗試 merge + State.set，可能互相覆蓋。 建議：先 syncDownloadAll 完成，再 startWatchers。
P1 4. 錯誤處理不一致
三種模式混用：(1) try-catch + console.error；(2) Events.emit(ERROR)；(3) 兩者都有。 部分 catch 只記錄不通知用戶，用戶完全不知道操作失敗了。 建議：統一為 try-catch + console.error + Toast.error 三件套。
P1 5. closedMonths 用陣列而非物件
watchers.js 第 163 行：用 JSON.stringify 比較陣列，效能差且順序敏感。 建議：改用物件 { "2026-06": true } 或 Set，比較更可靠。
P2 6. bridge.js 過於龐大
所有全域 onclick 函數集中在一個文件，雖然職責明確但超過 800 行不易維護。 建議：按功能拆分為 booking-bridge / agent-bridge / sync-bridge / misc-bridge。
P2 7. 缺乏 calc 層單元測試
build.js 有 95 項整合測試，但 calc/stats 和 calc/filters 的純函數沒有獨立單元測試。 建議：加入邊界條件測試（空陣列/null/undefined/極端值）。

三、同步功能

✓ 現有優勢

完整的 7+1 watcher 覆蓋
bookings/hotelConfig/agentList/employeeList/archives/closedMonths/settings + clearedAt
墓碑機制 + 30天TTL + recentlyDeleted 防復活
三重刪除保護，確保刪除操作正確同步且不會被舊數據覆活
時間戳衝突解決
_updatedAt 較大者勝出，含墓碑在內
重連自動同步
連線恢復後 3 秒 syncDownloadAll → 5 秒 syncUploadAll

待修復問題

P0 1. syncDownloadAll 不下載 settings
watchers.js 第 233 行：pending = 6， 6路並行下載只包含 bookings/hotelConfig/agentList/employeeList/archives/closedMonths， 缺少 settings。雖然 _watchSettings() 會處理，但如果 watcher 尚未觸發， settings 的初始值只有空物件 {}，roomFeeRate 會用預設值。 建議：加入 settings 下載分支，pending 改為 7。
P0 2. settings watcher 不 emit 事件
watchers.js 第 178-197 行：settings 同步後只 State.set('settings', remote)， 沒有 emit 任何事件。而 State.set() 會 emit UI_RENDER， 但 app.js 的 UI_RENDER handler 會重渲染當前頁面——如果當時不在需要顯示 roomFeeRate 的頁面，就不會更新。 建議：加 Events.emit(EVENTS.SYNC_SETTINGS, remote) 讓費用頁能監聽。
P0 3. _watchClearedAt 比較邏輯有誤
watchers.js 第 205 行：用 LAST_SYNC_TIME 而非專用的 LAST_CLEARED_AT 來比較 clearedAt 時間戳。 正常同步也會更新 LAST_SYNC_TIME，可能導致清除信號被「已同步」信號覆蓋而漏觸發。 建議：使用獨立的 Store.load('bk_last_cleared_at') 來追蹤。
P1 4. agentList/employeeList 不在 syncUploadAll
uploader.js 第 117-129 行：syncUploadAll 只上傳 bookings/archives/hotelConfig/closedMonths/settings， 不含 agentList 和 employeeList。如果離線時新增了代理，重連後只靠 watcher 觸發——但 watcher 是下載方向，不會主動上傳。 建議：在 objectPaths 中加入 agentList 和 employeeList。
P1 5. botLogs 無清理機制
syncBotLogToFirebase 只增不刪，botLogs 在 Firebase 中會無限增長，最終拖慢 watcher 下載速度。 建議：Bot 端每日清理 30 天以上的日誌，或 Web 端定期清理。
P2 6. 離線編輯隊列非持久化
uploader.js 的 _queue 在記憶體中，頁面刷新即丟失。 如果 Firebase 斷線時做了操作，刷新頁面後這些操作就不會上傳。 建議：將隊列持久化到 localStorage，刷新後恢復。
P2 7. transaction 失敗無重試
_uploadArrayPath 的 transaction 失敗只記錄 error，不重試。 建議：加入 1-2 次重試 + 指數退避。

768px 平板 + 480px 手機雙斷點，底部導航+抽屜側欄
豐富的組件庫
KPI 卡片/徽章/骨架屏/進度度條/圓環圖/矩陣表/空狀態等
列印樣式支援
隱藏交互元素，保留內容可列印

可優化項

P1 1. 總覽頁缺乏「今日提醒」視覺區塊
Bot 有今日入住提醒，但 Web 總覽頁沒有對應的視覺卡片。建議在 KPI 卡片和狀態分佈之間， 加入「今日入住 / 今日退房 / 待確認 / 待收費」四格提醒橫條，用鮮明色彩突出。
P1 2. 費用分佈圓環圖無動畫
overview.js 的 _feeDonut() 用 conic-gradient 靜態生成， 缺少載入動畫。建議加入 CSS transition 或 @keyframes 讓圓環從 0% 生長到目標百分比。
P1 3. 空狀態設計不一致
部分頁面用 _emptyStateMini()（純文字），部分用更精緻的空狀態（圖標+文字+按鈕）。 建議統一為一個 EmptyState 組件，含圖標、標題、描述、可選按鈕。
P1 4. 表格桌面版列數多時擁擠
費用收取頁有 13 列，在較小桌面螢幕上可能出現水平滾動。建議考慮： (1) 欄位寬度用 min-width 而非固定 width；(2) 部分次要欄位可收合/展開。
P2 5. 顏色對比度可能不足
--text-muted: #94a3b8 在 --bg-elevated: #ffffff 上的對比度約 2.8:1， 低於 WCAG AA 標準的 4.5:1。KPI 卡片的 kpi-sub 文字使用此色，在強光下手機可能難以辨識。 建議：將 muted 提深到 #64748b（slate-500，對比度 4.6:1）。
P2 6. 狀態分佈進度條首次渲染不觸發動畫
_statusRow() 的進度條有 transition: width 0.6s， 但首次渲染時 width 直接設為目標值，不會從 0 過渡。 建議：先設 width:0，然後用 requestAnimationFrame 設為目標值。
P2 7. 頁面切換無過渡動畫
切換頁面時內容直接替換，缺乏淡入效果。建議加 animation: fadeIn 0.2s。
P2 8. 列印樣式可改進
列印時仍顯示部分操作按鈕。建議在 @media print 中加 .btn, .page-actions { display: none !important; }。
📋 建議實施順序：
第一批（P0，立即修）：同步#1 settings 下載 + 同步#2 settings 事件 + 同步#3 clearedAt 比較 + 結構#1 State debounce + 結構#2 _hasChanged 修復

第三批（P1）：功能#4-7 + 結構#3-5 + 同步#4-5 + 美工#1-4
